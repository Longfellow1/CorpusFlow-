# CorpusFlow PRD Spec

## 1. Product Positioning

CorpusFlow 是一个面向评测 query 与微调数据生产的可控语料生成工具。

首期目标不是完整做成“诊断式增强平台”，而是先把两条主链路跑稳：

- 高级编辑模式
- 快速任务模式

产品核心价值不是黑盒生成，而是：

- 过程可控
- 可人工干预
- 生成自然
- 支持批量产出

---

## 2. Phase Scope

### 2.1 In Scope

- 高级编辑模式
- 快速任务模式
- 评测任务
- 微调任务
- 最小登录与个人任务隔离
- 前后端分离
- 后端统一接管生成、任务、导出、持久化

### 2.2 Out of Scope

- Bad Case 诊断式增强
- 多轮对话能力正式产品化
- 多人协作
- 复杂权限
- 复杂评测平台
- 多服务分布式架构

---

## 3. User Priority

首期用户优先级：

1. 测试同学
2. 算法工程师
3. 产品同学，用于补充边界 case

---

## 4. Core Product Model

任务由两层定义：

- 业务类型：`评测` / `微调`
- 操作模式：`快速任务` / `高级编辑`

这两层共同决定：

- 生成策略
- 预览方式
- 导出结构
- 质量控制策略

---

## 5. Mode Definition

### 5.1 高级编辑模式

定位：

- 质量优先
- 可控优先
- 适合精细生成、预览、人工修订

首期核心价值：

- 控制仿写/泛化幅度
- 避免明显偏题和过度发散
- 保持 seed 语义骨架稳定

### 5.2 快速任务模式

定位：

- 吞吐优先
- 弱编辑
- 面向大批量生成

特点：

- 允许一次输入大量 seed
- 鼓励多生成
- 允许一定比例损耗
- 过滤强度和通过率可调

---

## 6. Core User Journey

### 6.1 高级编辑模式

```text
创建任务
  -> 输入 seed
  -> 查看解析结果
  -> 调整任务级与 seed 级控制条件
  -> 点击生成预览
  -> 查看候选结果
  -> 选择或继续调整
  -> 正式批量生成
  -> 导出
```

### 6.2 快速任务模式

```text
创建任务
  -> 批量输入或上传 seed
  -> 设置生成倍数 / 过滤强度 / 双路径比例
  -> 直接批量生成
  -> 查看结果总量与通过率
  -> 导出
```

---

## 7. Product Principles

### 7.1 过程可控

系统必须让用户理解：

- 当前任务在做什么
- 生成结果从哪里来
- 可以如何调整

### 7.2 自然优先

首期不采用明显规则拼接感强的生成方式作为主方法。

### 7.3 宽松生成，轻量筛减

首期不依赖重门控，而采用：

- 前置拆解与骨架稳定
- 中段鼓励模型多生成
- 后段轻量筛减

### 7.4 经济账内生

系统需要考虑：

- token 开销
- 生成耗时
- 吞吐量

但产品策略不应因此把生成流程收得过死。

---

## 8. Algorithm Product Strategy

首期算法采用：

- P0 语义拆解
- L1 稳定骨架
- L2 双路径生成
- L3 轻量后处理

### 8.1 P0 语义拆解

目标：

- 把 seed 拆准
- 为后续生成建立稳定输入

### 8.2 L1 稳定骨架

目标：

- 稳住 seed 的核心语义
- 不走模板化笛卡尔乘积主路径

### 8.3 L2 双路径生成

路径分为：

- 常态泛化
- 鲁棒增强

这两个路径的比例由产品侧一个简单拖动条控制，默认偏向常态泛化。

### 8.4 L3 轻量后处理

只做：

- 格式过滤
- 明显异常过滤
- 轻量相似度去重

不做：

- 重型语义审判
- 多层 LLM 复判

---

## 9. Prompt Product Rule

产品必须保证：用户在前端输入和调整的每一个有效控制条件，都会在点击“生成预览”或“正式生成”时进入最终 prompt。

这包括：

- seed 原文
- 任务级整体要求
- 风格调整要求
- 多轮上下文要求
- 每条 seed 的解析结果
- 每条 seed 的人工修改结果
- seed 级补充控制信息

Prompt 不是后端内部细节，而是产品控制能力的核心实现。

---

## 10. Export Rule

导出结果由业务类型决定：

- `评测` -> 评测 query 数据
- `微调` -> 训练格式数据

两类导出首期同等优先级。

---

## 11. Success Criteria

### 11.1 Primary Objective

生成结果应能直接进入评测或训练流水线，并且内容本身可用。

### 11.2 Quality Strategy

质量判定采用：

- 自动规则筛选
- 人工抽检

自动筛选优先级：

- P0：语气 / 风格符合任务预期
- P1：重复或近重复
- P1：格式合规与字段完整

### 11.3 Core Metrics

- 内容可用性
- 去重率
- 数据格式通过率
- 单任务平均产出时长
- 人工修改率

### 11.4 Runtime Visibility

这些指标需要轻量透出给用户：

- 已生成数量
- 保留数量
- 当前通过率
- 预计剩余时间
- 导出记录

---

## 12. Backend Responsibility

后端接管：

- 账号
- 任务
- 生成
- 轻量过滤
- 导出
- 持久化

前端继续承担：

- 交互
- 展示
- 人工调整
- 结果查看

---

## 13. PRD Conclusion

首期 CorpusFlow 定义为：

一个保留现有 UI 形态、以后端正式接管为目标的可控语料生产工具。它通过高级编辑模式和快速任务模式两条主链路，为评测和微调场景提供自然、可控、可批量的 query 与数据生成能力，并通过提示词工程、稳定骨架和轻量筛减在质量、吞吐、成本之间取得平衡。
